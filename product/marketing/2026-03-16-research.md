## Reddit r/openclaw — Notable threads

**1. "I've helped 50 people debug their OpenClaw, these are the 5 things I always find" (312 upvotes)**
Someone who has reviewed 50+ OpenClaw setups across DMs, Reddit, and Discord distills the five most common and fixable problems:

1. Opus as the default model (10-15x cost for tasks where sonnet is indistinguishable)
2. Never starting a fresh session — "every message in your current session gets sent with every new API call." Fixing this alone cut costs 40-60% for three people they helped.
3. Installing skills without reading the source (malicious skills, context-bloating skills, looping cron skills)
4. Gateway exposed to the network (0.0.0.0 binding)
5. Adding a second agent before the first one works

Key quote: "People optimize for capability before stability. They want their agent to do more before it reliably does anything."

This is high signal. The session context accumulation problem (#2) is specific, real, and not obvious to most users. It's also the dimension where structural isolation (ManyClaw) answers what discipline (/new commands) only patches.

**2. "OpenLobster — Looking for contributors" (new/active)**
OpenLobster is a Go-based fork of OpenClaw. Their positioning is a clean articulation of OpenClaw's architectural limits:
- "Memory is a proper graph database — Neo4j or a local GML file — where the agent builds typed relationships over time, not a markdown file that breaks when two users write to it simultaneously."
- "Multi-user is first class: each person gets their own conversation history, memory, and tool permissions across every channel, without stepping on anyone else."
- "The scheduler is an actual scheduler — cron expressions and ISO 8601 datetimes, with status and logs visible in the dashboard, not a daemon reading a checklist file every 30 minutes."

This is useful framing. The community is aware of OpenClaw's seams. ManyClaw sits in this same conversation.

## Key themes today

1. **Session context accumulation is a real, underappreciated cost driver.** The 50-people post surfaces it clearly. Most users don't realize every message in their session history is re-sent on every API call. Cutting this is the highest-leverage single fix for most setups. ManyClaw's structural angle: you don't need discipline if isolation is architectural.

2. **The multi-agent architectural conversation is maturing.** OpenLobster's articulation of OpenClaw's limits is sharp. The community is naming the seams, not just working around them.

3. **Capability vs. stability tension.** People add complexity before the basics work. Relevant to anyone thinking about when multi-agent is actually the right move.

## Topic choice for today's post

**Winner: The session bloat problem**

The 50-people post surfaces something specific: context accumulation silently inflating costs. The fix the author recommends (/new before heavy tasks) is correct but requires ongoing discipline. The structural answer — separate agents for separate concerns, each with their own context — is what ManyClaw does. This is a real community pain point with a clear outside-in entry point and a natural ManyClaw connection that doesn't require any pitch language.

Angle: Start with the problem as the community names it. Explain what's actually happening. Show why discipline is a leaky fix. Land on structure as the durable answer.
